# 辞書

AIMO辞書は、タクソノミー内のすべての有効なコードの正式なリストです。各コードのラベル、説明、ライフサイクル情報を含む完全な定義を提供します。

## 目的と監査の文脈

辞書は以下のための**機械可読参照**として機能します：

1. **証跡テンプレート**: コードはEVテンプレートでAIシステムを分類するために使用されます
2. **バリデータ**: バリデータはすべてのコードが辞書に存在することをチェックします
3. **カバレッジマップ**: コードにより外部フレームワークや規制へのマッピングが可能になります

!!! info "SSOTの原則"
    辞書CSV（`taxonomy_dictionary_v0.1.csv`）はすべてのコードの唯一の正（Single Source of Truth）です。ドキュメントページ（このページを含む）は派生した説明です。疑問がある場合はCSVを参照してください。

## 辞書フォーマット

辞書は**21列**を持つCSVファイルとして配布されます：

| 列 | 必須 | 説明 |
| --- | --- | --- |
| `standard_id` | はい | 標準識別子（AIMO-STD） |
| `standard_version` | はい | 標準のバージョン |
| `dimension_id` | はい | 2文字の次元ID（FS, UC, DT等） |
| `dimension_name_en` | はい | 英語の次元名 |
| `dimension_name_ja` | はい | 日本語の次元名 |
| `code` | はい | 完全なコード（例：UC-001） |
| `label_en` | はい | 英語ラベル |
| `label_ja` | はい | 日本語ラベル |
| `definition_en` | はい | 英語定義 |
| `definition_ja` | はい | 日本語定義 |
| `scope_notes` | いいえ | 使用範囲に関する補足 |
| `examples` | いいえ | パイプ区切りの例 |
| `status` | はい | active, deprecated, removed |
| `introduced_in` | はい | コードが追加されたバージョン |
| `deprecated_in` | いいえ | 非推奨化されたバージョン |
| `removed_in` | いいえ | 削除されたバージョン |
| `replaced_by` | いいえ | 非推奨時の代替コード |
| `backward_compatible` | はい | 変更が後方互換かどうか |
| `references` | いいえ | 外部参照 |
| `owner` | はい | 責任者 |
| `last_reviewed_date` | はい | 最終レビュー日（YYYY-MM-DD） |

## CSV列定義

### 識別フィールド

- **standard_id**: AIMO Standardの場合は常に`AIMO-STD`
- **standard_version**: SemVer形式（例：`0.1.0`）
- **dimension_id**: 大文字2文字（FS, UC, DT, CH, IM, RS, OB, EV）
- **code**: フォーマット`<DIM>-<TOKEN>`（例：`UC-001`）

### ラベルと定義フィールド

- **label_en / label_ja**: 短いラベル（最大50文字推奨）
- **definition_en / definition_ja**: 完全な定義（1〜2文）
- **scope_notes**: コードをいつ使用する/しないかの補足
- **examples**: パイプ区切りの例（例：`chatbot|recommendation`）

### ライフサイクルフィールド

- **status**: コードの現在のステータス
  - `active`: 有効で使用中
  - `deprecated`: 有効だが削除予定
  - `removed`: 無効
- **introduced_in**: コードが最初に追加されたバージョン
- **deprecated_in**: 非推奨化されたバージョン（非推奨でない場合は空）
- **removed_in**: 削除されたバージョン（削除されていない場合は空）
- **replaced_by**: 非推奨時の代替コード
- **backward_compatible**: 変更が既存の使用を破壊しない場合`true`

### ガバナンスフィールド

- **references**: 参照する外部標準または文書
- **owner**: 責任者（例：`AIMO WG`）
- **last_reviewed_date**: 最終レビュー日（YYYY-MM-DD）

## コードの使用方法

### 証跡テンプレートでの使用

各EVテンプレートには8次元コードテーブルが含まれます：

```markdown
## AIMO Codes（8次元）

| 次元 | コード | ラベル |
| --- | --- | --- |
| **FS** | `FS-001` | 社内生産性 |
| **UC** | `UC-001`, `UC-002` | 一般QA、要約 |
| ...
```

### バリデータでの使用

バリデータは以下をチェックします：

1. 証跡で参照されるすべてのコードが辞書に存在する
2. コードフォーマットが期待されるパターンに一致する
3. 非推奨コードは警告を発する
4. 削除されたコードは拒否される

### カバレッジマップでの使用

コードにより外部フレームワークへのトレーサビリティが可能になります：

```
UC-001（一般QA） → ISO/IEC 42001:2023 A.5.2.1
DT-004（個人情報） → GDPR 第4条(1)
```

## 後方互換性ポリシー

`backward_compatible`フィールドは移行をガイドします：

| フィールド値 | 意味 | 監査人のアクション |
| --- | --- | --- |
| `true` | 既存の証跡は有効なまま | アクション不要 |
| `false` | 既存の証跡は更新が必要な場合がある | 影響を受ける証跡をレビュー |

### バージョン互換性マトリックス

| 証跡バージョン | 辞書バージョン | 互換性 |
| --- | --- | --- |
| 0.1.x | 0.1.x | 完全 |
| 0.1.x | 0.2.x | 完全（新コード追加、削除なし） |
| 0.1.x | 1.0.x | レビュー必要（破壊的変更の可能性） |

## 監査確認ポイント

証跡をレビューする際、監査人は以下を確認すべきです：

| チェック | 確認事項 |
| --- | --- |
| **定義の明確性** | `definition_en/ja`はコードの意味を明確に説明しているか？ |
| **ステータスの有効性** | コードは`active`か（`deprecated`や`removed`ではないか）？ |
| **バージョン整合性** | `introduced_in` ≤ 証跡の`taxonomy_version`か？ |
| **完全性** | すべての必須次元がカバーされているか？ |

## 拡張ガイドライン

組織は以下のルールに従ってカスタムコードで辞書を拡張できます：

### 拡張プレフィックス

カスタムコードは組織固有のプレフィックスを使用すべきです：

```
X-<ORG>-<DIM>-<TOKEN>
```

例：ACME社のカスタムユースケースコードとして`X-ACME-UC-901`。

### 拡張要件

1. カスタムコードは標準コードと競合してはなりません
2. カスタムコードはローカル拡張辞書に文書化されるべきです
3. 拡張は標準化の可能性のためAIMO WGに提出されるべきです
4. 外部関係者とエビデンスを交換する際は、標準コードのみを使用してください
